# NOTTE DELL'ORDA — guida per l'agente

Zombie shooter 3D a ondate. **Three.js 0.169** (vanilla, no framework) + **postprocessing 6.x**, build con **Vite**. Twin-stick dall'alto con visuale **prima persona** opzionale (tasto V). Tutto il codice e la UI sono **in italiano**: mantieni questa convenzione (testi UI e commenti in italiano).

## Avvio / dev
- `npm run dev` → server su **porta 3210** (`vite.config.js`, `strictPort`: fallisce, non cambia porta). La 5173 è di un altro progetto.
- Entry: `index.html` → `<script type="module" src="/src/main.js">`. HUD/menu/pausa sono HTML+CSS dentro `index.html`, pilotati da `src/ui.js`.
- Pagine dev secondarie (rotte via middleware in vite.config): **`/audios`** (banco prova suoni) e **`/models`** (visore GLB PBR). Build multi-input in `vite.config.js`.
- `npm run build` / `npm run preview` (preview su 3211).

## Compressione asset (caricamento veloce)
- Gli asset pesano ~1/3 dell'originale grazie a **`tools/compress-assets.mjs`** (usa `@gltf-transform/cli`, devDep, + `ffmpeg` per le texture). Comprime modelli (meshopt geometria + webp texture @1024, **niente** simplify/join → rig e animazioni intatti) e texture PBR standalone (`.jpg`→`.webp` @1024). `node tools/compress-assets.mjs` (tutto), `--models`/`--textures`, o un filtro per nome (es. `aiden`).
- **Originali in `public/assets/_orig/` (gitignorato)**: il tool legge SEMPRE da lì, quindi è ri-eseguibile senza degrado. Per ri-comprimere un asset basta cancellare il suo file in `_orig/` e rilanciare.
- ⚠️ **NON quantizzare modelli da cui si legge geometria precisa**: glock, **KRISS e Mark 23** sono `{ noQuant: true }` (solo webp, niente meshopt) perché `player._computeMuzzleLocal` legge i vertici in bind-pose per la bocca della canna — la quantizzazione meshopt sposta la bind-pose → bocca/flash/proiettili fuori posto (FPS e top-down). Il **Remington 870 è STATICO** (nessun rig): la bocca è dalla bounding-box, robusta → meshopt pieno. Gli altri skinnati (player/zombi) sono quantizzati senza problemi (renderizzati dalle ossa).
- ⚠️ **Ferma il dev server prima di ri-comprimere**: Vite/sirv tiene un handle sui file statici serviti → `rename` fallisce con `EPERM` su Windows. Il tool ha un retry, ma se persiste è il server (o un Chrome dei test) che blocca il file.
- Tempo a menu sceso ~70% (bench): byte eager ~40→16 MB, totali 88→25 MB. Bench: `tools/load-phases.mjs` (fasi assets/world), `tools/load-bench.mjs` (byte), `tools/model-verify.mjs <nome>` (un modello in /models), `tools/game-verify.mjs` (integrità in partita).

## Mappa dei file (`src/`)
- `main.js` — entry: renderer, postprocessing, stati di gioco (`menu|playing|paused|dying|gameover`), camera (top-down/FPS), loop ondate, punteggio, **valuta Anime**, **interazione porte (E)**, wiring opzioni. Oggetto centrale **`game`** (esposto `window.__game`).
- `config.js` — TUTTE le costanti: `CONFIG`, `WEAPONS`, `ENEMY_TYPES`, `WAVE_THEMES`, `BOSSES`, `DIFFICULTIES`/`setDifficulty`, `ZONES`, `SOULS`/`soulsFor`, `depthMods`.
- `world.js` — costruisce hub (cimitero) + **3 zone sbloccabili** + porte + atmosfera (luna, stelle, nebbia, lanterne, lucciole, pioggia-base). **Alberi morti = procedurali ramificati** (cilindri fusi con `mergeGeometries`, corteccia PolyHaven riusata da `barkMat`), **staccionata = ferro battuto** (pannello fuso `IRON_PANEL` + pilastri di pietra). Espone `rooms`, `gates`, `confine`, `unlockZone`, `resetZones`, `nearestGate`, `atmoFog`/`atmoDensity`.
- `player.js` — movimento, mira, 4 armi, ricarica, scatto (con scia-fantasma), **viewmodel FPS**, mira procedurale di braccio/testa **clampata** rispetto al busto, proiettili. **Passi sincronizzati alla FASE dell'animazione** (2 appoggi/ciclo via `_stepHalf`, non a tempo fisso). **Le 4 armi sono modelli realistici skinnati con clip Reload/Shoot/Hide/Draw**: Glock-17, KRISS Vector (mitra), Mark 23 (magnum) — stessa "serie BarcodeGames" → **mani guantate identiche** e clip uguali; il Remington 870 (pompa) è **statico** e al mount **prende in prestito le mani guantate ANIMATE del Glock** (`_mountGun`: se l'arma non ha mani proprie, costruisce il rig del glock con l'arma nascosta e ne usa le clip → la ricarica muove le mani). **`_gunMixer`+`_gunReload`+`_gunShoot` valgono per TUTTE le armi**; `_trimClip` estrae la finestra reale dei keyframe di ogni clip (le clip stanno su una timeline condivisa: es. lo Shoot del glock è ai frame ~98-109) e la ribasa a 0 → niente più frame hardcodati. ⚠️ **Flag per-arma nel MANIFEST** (`assets.js`, propagati in `Assets.guns`): **`axis:'z'`** = canna già lungo +Z (l'euristica "asse più lungo" sbaglia sulle pistole tozze: il Mark 23 è più ALTO che lungo → ruotato a sproposito, canna in su). **`flip:true`** = modello con la canna verso -Z → girato 180° (il Remington 870 sennò **spara all'indietro**, sia FPS che top-down). **`gloveOnly:true`** = nasconde l'avambraccio di pelle e tiene solo il guanto (sui modelli più scalati l'osso aperto del gomito finiva davanti alla camera → "braccio tagliato"; il KRISS lo usa). **`borrowHands:true`** = nasconde le mani proprie e usa quelle del Glock (il Mark 23 è una pistola → mani+ricarica del Glock calzano, mentre il suo guanto in viewmodel mostrava un taglio). **`handGrip`** = ritocco scala/posizione delle mani in prestito. **`vmShift`** = sposta il viewmodel nel frame camera (il KRISS è abbassato così il gomito "tagliato" del braccio esce sotto lo schermo). ⚠️ **`viewmodel:true`** (fucile a pompa Cransh) = rig FPS COMPLETO braccia+arma → mount dedicato `_mountViewmodel`: la **bind-pose è "esplosa"** (coordinate enormi, inutilizzabili), quindi misura la geometria **in posa idle** con `SkinnedMesh.computeBoundingBox()` (vertici deformati), scala la canna a `length`, classifica canna (più lunga in Z) vs braccia (più larghe in X → nascoste in top-down), calcola la bocca, fa girare l'**idle in loop** (sparo/ricarica una volta poi `_playIdle` via evento `finished` del mixer). **Ricarica a COLPO SINGOLO** (`WEAPONS.shotgun.shellReload`+`shellTime`): `startReload` avvia `_shellReloading`, l'update carica **un bossolo ogni `shellTime`s** (`mag++`, `reserve--`) fino al massimo; **sparare la ANNULLA** tenendo i bossoli già caricati (vedi blocco fuoco: `if (_shellReloading && mag>0) { _endShellReload(); _fire() }`). L'animazione (clip ~6.8s) scorre su `mag*shellTime`. `vmAdjust` posiziona il modello dentro il gunMount; `shootFit` = durata dello sparo (più lenta del carrello pistola).
- `enemies.js` — `Enemy` (IA inseguimento+separazione, spawn dal terreno, varianti, boss con abilità) + `WaveDirector` (ondate, spawn, sputi acidi). Esporta `resolveCollisions` e `setConfine`. Brute/boss umanoidi = **Mutant realistico** (`mutant`, 13 anim Mixamo); walker = **Aiden** (zombie realistico Sketchfab CC-BY, idle/walk/attack); runner = **Larnox** (Sketchfab CC-BY-NC, 9 anim: corsa/attacchi/morte/urlo); crawler = **Wolf** (Sketchfab, cane che carica, anim run/creep); Hazmat+scheletri = varianti tardive. I modelli Sketchfab si scaricano via Download API ufficiale (`tools/sketchfab-dl.mjs`, serve un API token); hanno root motion → `stripPos:true` nel MANIFEST toglie le tracce di posizione (sennò scivolano). Attacco = clip + **affondo** in avanti. La **carne dei nemici è forzata OPACA** (roughness ≥0.9, metalness 0 in enemies.js) sennò la luna la fa sembrare bagnata/plastica. **Solo il walker dell'ondata 1 (`zombie_aiden`) è eager**; TUTTI gli altri nemici sono `deferred:true` (caricati in sottofondo dopo il menu) — il primo nemico spunta ≥2.4s dopo "GIOCA", quindi arrivano in tempo senza rallentare l'avvio (catena di ripiego/procedurale se in ritardo). Shader pre-scaldati **incrementali e fuori dal percorso critico** (`prewarmShaders` in main.js: un modello per frame via rAF, NON blocca il menu) → niente scatto al primo spawn ma menu reattivo subito. ⚠️ Il prewarm fa un **render vero fuori schermo** (su `_warmRT`, con `shadowMap.needsUpdate`), NON solo `renderer.compile`: quest'ultimo linka solo il programma principale, mentre il **depth program del caster d'ombra** (skinned) lo compila three solo al primo render della shadow map — da lì il freeze di 1-2s al primo spawn se non lo si scalda.
- `effects.js` — pool di particelle (additive/alpha), muzzle flash, traccianti, **sangue (schizzo direzionale all'impatto, NIENTE pozze/decal a terra)**, onda d'urto/detriti, dash, anelli, numeri danno, luci da pool. Niente alloc per-particella nelle hot path.
- `assets.js` — caricamento GLB (giocatore/zombi/armi/props) + texture PBR; `Assets`, `Animator` (matcher clip per scopo), fallback procedurali. Texture realistiche in `Assets.tex` (incl. `hubGround`). Props realistici PBR PolyHaven in `assets/models/ph/` (tronchi/rocce/ceppi/busto/lanterna, ora **`.glb` autosufficienti** compressi); lapidi/mausolei/plinti sono geometria PBR procedurale in `world.js`. **Tutti gli asset sono compressi** (vedi sotto): GLB con `EXT_meshopt_compression` (geometria) + `EXT_texture_webp` (texture), texture standalone in `.webp`. Ogni set PBR è scaricato/decodificato **una sola volta** e clonato per i `repeat` diversi (`_pbrCache` in `loadPBRSet`) → niente doppio decode (forest serve a `forest`+`hubGround`, rock a `rock`+`graveStone`).
- `audio.js` — WebAudio: bus master/musica/sfx + riverbero a convoluzione; `Audio.play(name)` usa il **file** se la chiave è in `AUDIO_MANIFEST`, altrimenti il **synth**.
- `pickups.js` — medikit/munizioni/armi (alone sprite additivo, NO luci dinamiche).
- `input.js` — tastiera/mouse/rotella + mouse-look in pointer lock (`lookDX/DY`).
- `ui.js`, `rain.js`, `audios.js`/`models.js` (pagine dev).

## ⚠️ Trappole che fanno fallire — leggi PRIMA di toccare
- **NON aggiungere/rimuovere `THREE.PointLight` a runtime**: forza la ricompilazione di TUTTI i materiali → scatto vistoso. Le luci si creano una volta (build) e si **modula solo `.intensity`**. I pickup usano sprite additivi, non luci. Le luci delle zone sono create al build (sempre presenti, anche a zona chiusa).
- **Postprocessing**: Bloom e ChromaticAberration sono entrambi effetti di convoluzione → vanno in **EffectPass SEPARATI**, altrimenti errore runtime "Convolution effects cannot be merged" (vedi `main.js`).
- **Freeze al primo spawn = shader d'ombra**: `renderer.compile` NON compila il depth program del caster d'ombra skinnato (lo fa three al primo render della shadow map) → 1-2s di freeze quando il primo nemico compare. `prewarmShaders` deve fare un **render reale fuori schermo** con `shadowMap.needsUpdate=true`, non solo `compile`. Vale per qualunque nuovo modello skinnato che proietta ombra.
- **GLTFLoader DEVE avere `setMeshoptDecoder`**: ORA **tutti** i modelli usano `EXT_meshopt_compression` (post-compressione), altrimenti non caricano (`assets.makeLoader`, `models.js`). Il webp (`EXT_texture_webp`) è nativo in GLTFLoader, nessun setup extra.
- **Viewmodel FPS in WORLD-SPACE, NON figlio della camera**: parentare alla camera NON lo renderizza (il renderer percorre la scena, non i figli della camera). Vedi `player._updateViewmodel`: base ortonormale dello sguardo, l'arma orientata con `makeBasis` perché **+Z locale di `gunMount` = canna**. ⚠️ **CONVERGENZA sul mirino**: la base NON usa `F` puro (sennò la canna resta parallela allo sguardo e *spostata di lato* → non punta il mirino), ma un forward `cF` = (occhio + F\*30 − posizione arma): punta a un punto sul raggio di vista, che si proietta sul mirino → la volata "guarda" il mirino per **tutte** le armi. I proiettili partono comunque lungo `F` (`_fwd`). Il mirino è **circolare** (anello+punto, CSS `#crosshair` in index.html).
- **Mirino = cono di SPREAD, con rinculo (bloom)**: ogni arma ha `spread` (cono base, gradi) + `bloom` (gradi aggiunti a ogni colpo, `player._bloom`, che rientra con `damp(...,0,7)`). Lo **spread effettivo** = `player.currentSpread()`. `main.updateCrosshairSize` dimensiona il cerchio sulla proiezione di quel cono (**GAIN=1** → cerchio = cono) — **FPS**: proiezione angolare `(H/2)·tan(sp)/tan(fov/2)`; **top-down**: proiezione del raggio a terra alla distanza del cursore (`aimPoint.project(camera)`). I proiettili partono nello stesso cono → **cadono sempre dentro al cerchio**. Sparare → il cerchio cresce e poi rientra. ⚠️ **In FPS il mirino è un OGGETTO 3D nella scena** (`cross3dRing`/`cross3dDot`, sprite billboard a `CROSS3D_DIST=2.4` davanti alla camera con `depthTest:true`): così **l'arma lo OCCLUDE** (il mirino si vede *dietro* il fucile, non disegnato sopra come farebbe un overlay DOM). In **top-down** resta l'overlay DOM `#crosshair` (CSS in index.html) sul cursore; `updateCrosshairSize` toggla l'uno o l'altro per visuale e va chiamata **DOPO** l'aggiornamento camera (il mirino 3D usa posizione/sguardo del frame corrente, sennò lagga). Mirino volutamente **tenue** (valore basso, niente glow) per non innescare il **bloom** + contorno scuro per la leggibilità. ⚠️ I proiettili NON sono box/dardi spessi: **capsula SOTTILE** (`_bulletGeo`, opacità bassa) che traila DIETRO una **testa luminosa** piccola (sprite `softDotTexture`) + scia fine → tracciante discreto e realistico. Anche il tracer della pool (`effects.js`) è ora un **cilindro tondo** open-ended, non `BoxGeometry`.
- **Confine area di gioco**: NON è più un clamp radiale. È `world.confine` (unione delle stanze attive), iniettato in `enemies.js` via `setConfine(world.confine)` da `main.js` dopo `buildWorld`. `resolveCollisions` chiama `_confine`. Despawn proiettili/sputi usa `world.maxExtent`.
- **Mira braccio/arma**: in top-down i proiettili partono lungo `player._aimYaw` (angolo dell'arma **clampato** rispetto al busto), **NON** verso il cursore quando il cursore è oltre l'angolo massimo. Clamp **asimmetrico** (`MAX_ARM_OUT` ampio verso il lato dell'arma, `MAX_ARM_IN` stretto verso il petto, lato rilevato da `_gunSide`). La testa gira con `MAX_HEAD`.
- **Bocca dell'arma**: calcolata dalla geometria (`player._computeMuzzleLocal` → `_muzzleLocal`, esclude mani e caricatore, alza Y sulla linea della canna). `_gunPos` = quel punto in coord mondo. In FPS il proiettile parte ESATTAMENTE da lì.
- **Audio**: per cambiare/aggiungere un suono basta aggiungere la chiave + l'`.ogg` ad `AUDIO_MANIFEST` (zero modifiche ai call site). Tutti i suoni sono **CC0/Pubblico Dominio**. ffmpeg è in `C:\program files\ffmpeg\bin\` (mix/trim → ogg mono). Es. `zombie_hit` (impatto proiettile sulla carne) = 3 varianti mix CC0 (schizzo carnoso + tonfo grave + transiente), rigenerabili con `tools/make-zombie-hit.mjs` da `tools/_audiosrc/impacts/`.
- **Opzioni**: `game.opt = { blood, damage }`, persistite in `localStorage` (`noh_opt_*`). I toggle stanno in **un'unica scheda OPZIONI** (modale `#options`) apribile dalla **rondella** (`#btn-options` nel menu, `#btn-options-pause` in pausa); la rondella del menu compare solo a risorse caricate (`ui.readyToPlay`). Sincronizzati via `data-opt` (un solo set di checkbox ora). `enemies.js` legge `game.opt` a ogni colpo. I numeri danno sono **OFF di default**.
- **Volume**: master/musica/sfx persistiti in `localStorage` (`noh_vol-*`). La **MUSICA parte a 0** (muta) di default — `_vol.music=0` in `audio.js` e slider `#vol-music` a 0; l'utente la alza dalla scheda Opzioni.

## Sistemi chiave (in breve)
- **Anime (valuta)**: `game.souls`, guadagnata a ogni kill (`config.soulsFor`), separata dal punteggio/record. Spesa per aprire le porte (`game.tryUnlockGate`, tasto **E** vicino al cancello).
- **Zone**: hub centrale + Cripta/Chiesa/Bosco dietro porte. Aprire = `world.unlockZone` (attiva la room, apre il cancello, espande l'area). Più zone aperte → `depthMods(game.zonesUnlocked)` alza hp/velocità/danno/densità/spawn (in `director.startWave`). Atmosfera della nebbia cambia per zona (`world.atmoFog/atmoDensity`, usata come base in `main.js`, il meteo modula sopra).
- **Varianti nemici**: ogni non-boss ha taglia/colore(HSL nella palette dell'ondata)/velocità/voce diversi + barcollio + ondeggio (`enemies.js` costruttore + `Animator.desync`).
- **Difficoltà**: `DIFFICULTIES`/`setDifficulty` (4 livelli) modulano nemici e soprattutto il **dash**.

## Test headless (puppeteer)
- `tools/*.mjs`: ognuno lancia Chrome di sistema con `puppeteer-core`. Esegui dalla **root del progetto** col dev server attivo (`node tools/<x>.mjs`).
- Flag obbligatori: `--no-sandbox --enable-unsafe-swiftshader --use-angle=swiftshader --mute-audio`. Path Chrome: `C:\Program Files\Google\Chrome\Application\chrome.exe`.
- **SwiftShader gira a ~1/6–1/8 del tempo reale** e `main.js` clampa `dt` a 0.05 → il **tempo di gioco scorre più lento del tempo reale**: dimensiona le attese generose, e i lerp framerate-dipendenti convergono lenti. Falsi negativi tipici: "reload non completata", "movimento FPS" (con pointer lock i `page.keyboard` possono non registrarsi → inietta `window.__game.input.keys.add('KeyW')`).
- Test utili: `zones-check`, `facing-check`, `fps-viewmodel-check`, `effects-check`, `variants-check`, `glock-check`, `fps-check`. Screenshot in `tools/_shots/` (gitignorato).
- Diagnostica esposta: `window.__game`, `__CONFIG`, `__audio`, `__GUNCAL` (calibrazione arma top-down), `__VM` (viewmodel FPS).
